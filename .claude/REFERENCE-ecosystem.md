# Ecosystem — Four-Component Composition

> Source: derived 2026-05-15 in a strategy conversation that began with "a good org mission for ServiceGrid" and recursed to the ecosystem level. A domain's mission is the reusable component-capability it exports to the level above; the ecosystem is the composition of all four.
> Revisit when: defining or scoping any of the four domains (SG / SO / DT / PG); evaluating whether a domain's work advances its exported component; reasoning about what the orchestrator substrate is for.

---

## 1. The four components

A domain's mission = **the reusable component-capability it exports upward** (build-around: solve it once, the ecosystem instantiates it).

| Domain | Exported capability | Autonomy layer |
|---|---|---|
| **ServiceGrid (SG)** | agents run a business | **resources** — pays for itself |
| **Superorganism (SO)** | an agent team (digital + physical) navigates any environment | **execution** — acts anywhere |
| **Dreamtime (DT)** | agents self-organize | **structure** — forms / restructures itself, no human architect |
| **Pilgrimage (PG)** | agents design & build any *finite* game that stays nested in, and serves, the infinite game — never usurping the frame | **purpose** — defines what it's doing, meaning intact |

Definitional clarifications (so the components don't bleed into each other):

- **SG** scales *horizontally* (instances). Current instantiation: field-service SaaS for the trades — this funds and proves SG; it is not a separate level.
- **SO** scales *vertically* (environments — one network that accretes capability per environment mastered). SG→SO bridge: SG's agent-direction capability (digital + physical, incl. robots / AR) transfers wholesale to SO, pointed at a person's environments instead of a business's operations. SO's environment-axis defines the space SG's instance-axis runs over.
- **DT** is *emergence* (the collective arranges and adapts itself) — distinct from SO, which operates a *given* configuration. Direction vs. self-formation.
- **PG**: the finite-nested-in-infinite clause is load-bearing — it is where meaning lives (see `~/.claude/CLAUDE.md` → "finite-infinite-game-framing"). A paperclip-maximizer is precisely a finite game that has usurped the infinite frame; correct game design structurally cannot emit one. The infinite game is by definition unspecifiable — it is the deliberately-unstated terminal. You do not flag it; you keep playing it.

## 2. The ecosystem goal (derived)

A self-running meaningful **mission** requires exactly these four — and the requirements *derive* the components, rather than the components being asserted:

- **fund** itself → SG
- **operate** itself in any environment → SO
- **organize** itself, no human architect → DT
- **define** what it pursues, meaningfully → PG

Remove any one and a human re-enters the loop at that layer. Composed:

> **A mission that funds, operates, organizes, and games itself — so human attention goes to the mission, not to running the thing that pursues it.**

"Missions fund & run themselves" is the economical handle; the precise goal is the four-part composition. It is an *ecosystem-level* property — it requires all four domains' exported components — which is why it sits inside no single domain.

The orchestrator is the shared substrate that composes the four: the proof this is one ecosystem, not four domains with a shared slogan. See [`ARCHITECTURE.md`](../ARCHITECTURE.md).
